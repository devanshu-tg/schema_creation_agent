# Multi-CSV demo dataset

7 tables totaling ~20k rows. Generated by `build/synthesize_multi_csv.py`
(seed 7777, reproducible).

## Relevant tables (5)

| Table | Rows | Joinable on |
|---|---|---|
| `customers.csv` | 2,000 | `customer_id` (PK) |
| `accounts.csv` | 3,000 | `account_id` (PK); `customer_id` (FK) |
| `merchants.csv` | 500 | `merchant_id` (PK) |
| `devices.csv` | 3,500 | `device_id`, `customer_id` (FK), `ip_address` |
| `transactions.csv` | 10,000 | `transaction_id` (PK); `account_id` (FK); `merchant_id` (FK) |

## Irrelevant tables (2)

These exist purely to test the agent's discovery-summary exclusion narrative.
Neither shares an identifier with the rest of the tables.

| Table | Rows | Why excluded |
|---|---|---|
| `internal_audit_log.csv` | 1,000 | Pure ops/audit log — no business entities |
| `marketing_campaigns.csv` | 200 | Standalone campaign metadata; no customer/account FKs |

## Embedded fraud signal

~60 customers form 12 small rings, each ring shares one device. Roughly
35% of transactions from ring accounts are flagged `is_fraud=1`, with
~0.5% baseline fraud rate across the rest. This gives the agent enough
shared-identifier signal to recognize a fraud-investigation shape and
to demonstrate ring-detection traversal.

## Expected agent behavior

When uploaded to a workspace and asked "find fraud rings":

1. `list_tables` returns 7 tables.
2. `summarize_discovery` returns 5 relevant + 2 excluded with reasons.
3. `match_all_patterns` ranks fraud-investigation first
   (10/10 or close — Customer, Account, Card-ish, Transaction, Merchant,
   Device, Email, Phone, Address; IPAddress from devices).
4. Agent narrates: *"I see 7 datasets. 5 look relevant; excluding
   `internal_audit_log` and `marketing_campaigns` because they don't
   share identifiers with the rest."*
